# Open Questions

## 設計上の未決定事項

### 1. qlib の時間軸と Flow Surface のティック軸の橋渡し

qlib は日次/分次のクロスセクション戦略が前提。Flow Surface の M1〜M15 リプレイで使えるのか、Q0 PoC で実測する必要がある。

- **検証ポイント**: Alpha158 のうち分次データで意味のある特徴量はどれか
- **Fallback**: 全 158 個ではなく、短期テクニカル系（RSI/MACD/ATR 系）のサブセット定義を作る

### 2. データ単位（Instrument）の扱い

qlib は「銘柄 × 日付」のロングフォーマットを前提。Flow Surface の単一銘柄リプレイでも `instrument` 列をダミーで作るか、複数銘柄同時にリプレイ可能にすべきか。

- **暫定方針**: Q1 では単一銘柄ダミー `instrument="SELF"` で進める
- **将来課題**: マルチ銘柄観測 API（`GET /api/replay/state?tickers=...`）の追加可否

### 3. reasoning 生成方式

- **A**: LightGBM 標準の `feature_importance`（軽量・高速）
- **B**: SHAP 値（解釈性が高いが重い）
- **C**: 両方をオプションで提供 ★暫定採用

ユーザー検証を経て主流派を決める。

### 4. モデル学習の場所

- **A**: 完全にユーザー側（SDK は predict のみ） ★現状方針
- **B**: SDK にトレーニングヘルパを同梱

Q1 では A、Q2 で B の必要性を再評価。

### 5. オフライン学習データの永続化

学習済みモデルをどこに置くか。

- 候補: `~/.flowsurface/models/qlib/` に joblib で保存
- ナラティブの `agent_id` と紐付け、リプレイ時に同一モデルが復元できるようにすべきか

### 6. 「Python 単独モード」との関係

Rust なし Python 単独モードが新設される将来、qlib 統合はそのモードでも動くべきか、Rust モード限定機能か。

- **暫定**: Python 単独モードでも動く前提で設計（HTTP API を呼ばない直接モードの分岐を qlib_adapter に持たせるかは Q3 で判断）

## 運用上の未決定事項

### 7. CI への組み込み

qlib は LightGBM 等の重い依存を持ち込む。CI 全ジョブに入れるか、qlib 専用ジョブとして切り出すか。

- **暫定方針**: 専用ジョブ `e2e-qlib` を追加。既存 headless ジョブには影響させない

### 8. ライセンス・帰属表示

qlib は MIT。`NOTICE` への記載および README 連携が必要かを法務（個人開発でも将来配布時のため）視点で確認。

### 9. Phase 4c（Ocean Protocol）連携の前倒し可否

qlib のバックテスト結果（Sharpe / 勝率）は Ocean のメタデータに直接流用できる。Phase 4c を qlib 統合と同時に PoC するか、ロードマップ通り Phase 4b 完了後に着手するか。

- **暫定**: ロードマップ通り。前倒しはしない。
