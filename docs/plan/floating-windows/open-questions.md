# Floating Windows 移行: Open Questions

## 着手前に確定すべき事項

### Q1. ドラッグ中の中間 `on_move` 発行頻度（未決定）

**背景**: `CursorMoved` は毎フレーム発行されうる。現設計では `MouseButtonReleased` 時のみ
`on_move` を発行してアプリ State 更新を抑えているが、これだとドラッグ中のパネルが
「ついてこない」見た目になる可能性がある。

**選択肢**:

| 案 | 概要 | メリット | デメリット |
|----|------|----------|----------|
| **A** | `MouseButtonReleased` 時のみ `on_move` を発行。ドラッグ中はウィジェット内部座標で描画 | State 更新ゼロ・再描画コスト低 | iced の `Widget::draw()` 内でドラッグ中座標を参照する必要があり、描画ロジックが分岐する |
| **B** | `CursorMoved` ごとに `on_move` を発行 | 実装シンプル | State 更新が毎フレーム走り、全ウィジェットが再描画される |
| **C** | `CursorMoved` ごとに発行するが、`Dashboard::update()` では `FloatingPane.rect` だけ更新して他の State を触らない | B より軽量 | `update()` の設計を慎重に行う必要がある |

**現時点のデフォルト**: 案 A で実装し、ジャーキーな見た目が問題になった時点で案 C に変更する。

---

### Q2. ズーム時のフォントサイズ・UI スケーリング（未決定）

**背景**: `Widget::draw()` はカメラの zoom 倍率を使ってパネルの位置・サイズをスケールするが、
パネル内部のフォントサイズや線幅を zoom に追従させるかどうかが未決。

**選択肢**:

| 案 | 概要 |
|----|------|
| **A** | 内部コンテンツのスケールは行わない（ズームはウィンドウ枠の拡縮のみ） |
| **B** | zoom に応じてフォントサイズ・アイコンサイズも比例スケール |
| **C** | zoom が 1.0 未満のときだけ内容を省略表示（ミニマップ的表現）にする |

**現時点のデフォルト**: 案 A（最小実装）で Phase 3 を完了し、Phase 5 以降で再評価。

---

### Q3. ウィンドウがスクリーン外にドラッグされた場合の挙動（未決定）

**背景**: カーソルをキャンバス外に移動させてもドラッグが継続できる場合、
パネルがスクリーン外（ネガティブ座標やキャンバス外）に出うる。

**選択肢**:

| 案 | 概要 |
|----|------|
| **A** | ワールド座標に制限を設けない（パンすれば辿り着ける） |
| **B** | タイトルバーが常にスクリーン内に収まるようドラッグ終了時にクランプ |
| **C** | ドラッグ中リアルタイムにスクリーン境界をクランプ |

**現時点のデフォルト**: 案 A（Figma 流）。Phase 4 の動作確認で使い勝手を見て判断。

---

### Q4. デフォルトレイアウト（`Dashboard::default()`）の座標算出方法（未決定）

**背景**: `Dashboard::default()` は初期ウィンドウサイズが確定する前に呼ばれる可能性がある。
`1280×800` をハードコードして比率計算するか、ウィンドウサイズ確定後に再計算するかが未決。

**選択肢**:

| 案 | 概要 |
|----|------|
| **A** | `DEFAULT_VIEWPORT_W = 1280 / DEFAULT_VIEWPORT_H = 800` でハードコード。実際のウィンドウサイズと少しズレてもズーム・パンで補正できる |
| **B** | ウィンドウリサイズイベント（`window::Event::Resized`）で初回のみ再配置する |

**現時点のデフォルト**: 案 A（最小実装）。saved-state のフォールバック時のみ発動するため精度より単純さを優先。

---

### Q5. `Widget::diff()` の実装方針（未決定）

**背景**: iced の `Widget::diff()` は子の数が変わるときに状態ツリーを正しく同期しないと
子ウィジェットの State がズレてパニックする。`FloatingPanes` は子数が動的に変わるため
特に注意が必要。

**調査事項**:
- `multi_split.rs` が `Widget::diff()` をどう実装しているか確認する
- 子が増減した際に古い `InternalState` が残らないかを検証する

---

### Q6. popout ウィンドウ内の FloatingPanes のカメラ初期状態（未決定）

**背景**: popout（OS 別ウィンドウ）内も `Vec<FloatingPane>` を使うが、
popout 固有の `Camera` を持つべきか、メインキャンバスの `Camera` と共有するかが未決。

**選択肢**:

| 案 | 概要 |
|----|------|
| **A** | popout は独自 `Camera`（デフォルト: `zoom=1.0, pan=(0,0)`）を持つ |
| **B** | popout はメインキャンバスの `Camera` をコピーして使う（起動時スナップショット） |

**現時点のデフォルト**: 案 A。popout は通常 1 パネルなのでズーム・パンを独立させた方が操作感が自然。

## 着手後に決めれば良い事項

- パネルのスナップグリッド（等間隔吸着）の必要性
- キーボードショートカットでのパネル間フォーカス移動（`Tab` / `Shift+Tab`）
- パネルの最小化（タイトルバーのみ表示）機能
- ズーム倍率・パン位置のリセットショートカット（`Ctrl+0`）
- パネル追加時のアニメーション（フェードイン）
- 複数パネル選択・一括移動の必要性
