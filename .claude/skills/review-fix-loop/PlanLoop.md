# PlanLoop — 計画書レビュー・修正ループ

計画書（`docs/` 配下の `.md` ファイル群）のレビュー・更新に使う。
共通ルール・収束基準は [`SKILL.md`](./SKILL.md) を参照。

---

あなたはオーケストレーターです。
計画文書群（`docs/<area>/*.md`）に対して「レビュー → 集約 → 統一方針決定 → 修正 → 検証」を 1 ラウンドとし、MEDIUM 以上の Finding がゼロになるまで反復させます。

> 使い方: ファイル末尾の「汎用呼び出しテンプレート」を新しい会話にコピーし、`{{}}` プレースホルダーを実値に置換して使う。汎用ロジックはこちら、案件固有情報はテンプレートで渡す。

## 収束の期待値（実測ベース）

中規模フェーズ（計画書 4〜6 ファイル・合計 500〜2000 行）の典型的収束カーブ:

| ラウンド | HIGH+MEDIUM 件数 | 説明 |
|---|---|---|
| R1（初回） | 25–40 | 用語不統一・フロー欠落・不変条件の矛盾が重複指摘される |
| R2 | 10–18 | 初回修正後、新記述が生んだ軽微な矛盾が中心 |
| R3 | 3–8 | 残存 MEDIUM、アンカー切れ・命名揺れ |
| R4（収束） | 0 | サニティチェックのみ |

**1 ラウンドで収束することはほぼない。3–4 ラウンドを見積もる。** HIGH/MEDIUM 件数が 3 ラウンド連続で横ばいなら観点設計が間違っているサインなので、ユーザーに相談する。

## 起動チェック

ラウンド 1 開始前に必ず以下を実行する:

1. 対象ディレクトリを `ls` し、計画文書ファイル一覧を確認
2. `review-fixes-<日付>.md`（修正ログ）の存在確認。**無ければラウンド 1 として新規作成する**。日付は `currentDate` を使う
3. 対象ファイル合計行数を `wc -l` で測り、レビュー範囲の規模感を把握（4000 行超ならレビュー観点を分割追加検討）
4. 直前 1〜2 コミットの差分を `git log --oneline -5` で確認し、計画変更の背景を把握

## ループ手順

### Step 1: レビュー（サブエージェント並列）

**4 観点（A/B/C/D）を 4 つの `general-purpose` サブエージェントに並列で割り当てる**。

4観点（A/B/C/D）の定義と各観点の追加チェック項目は `.claude/skills/e-station-review/SKILL.md` の「計画文書レビュー観点」セクションを参照すること。

**各サブエージェントへの共通指示**:

- Findings は **HIGH / MEDIUM / LOW** に分類。各件「ファイル:行範囲 / 問題内容 / 推奨修正方針」を添える
- **`review-fixes-<日付>.md` に既出の Finding は除外**（プロンプト本文で明示）
- **新規 Finding のみ報告**
- **語数上限を切る**（観点 A 400 語 / B 400 / C 400 / D 400 程度）
- 終盤ラウンド（HIGH/MEDIUM が減ってきた段階）では「**収束確認モード**: HIGH/MEDIUM ゼロなら『収束』と明記」を付ける
- 案件固有のスコープ外指摘（例: 他 venue 追加、別フェーズ巻き取り）を出さない旨を明記
- SKILL の一次資料を計画文書で上書きしない方針を明記

**並列起動は必ず単一メッセージ内の複数 `Agent` 呼出で行う**（直列起動は時間の浪費）。`run_in_background: true` 推奨。

### Step 2: 集約と統一方針決定（オーケストレーター本人）

サブエージェントの結果を読み終えたら、以下を**必ずオーケストレーター本人が決定する**:

1. **HIGH/MEDIUM/LOW を整理して件数を出力**（出力形式は「出力形式（毎ラウンド）」セクション参照）
2. **終了条件チェック**:
   - HIGH/MEDIUM ゼロ → ループ終了 → 「完了サマリ」セクション
   - 残存あり → 次ステップ
3. **統一方針（先決事項）の決定**:
   - 同一 Finding の解決策が複数ファイルに横串で波及するもの（用語統一・enum 値・HTTP ステータス選択・命名規約・ガード方針 など）について、**ラウンド内で 1 つの決定に集約**する
   - 例: `RATE_LIMITED` HTTP コードを 429 にするか 503 にするか / `order_number` を `venue_order_id` に統一するか / 引数追加 vs builder pattern
   - **これらの決定はテキストでまとめて表示**し、後続の implementer サブエージェント全員に同一文言で渡す（プロンプト先頭に「統一決定」セクションを置く）

> ⚠ **ここを飛ばすと修正サブエージェントごとにバラバラの判断をして次ラウンドで再分裂する**。今回のループ運用での最大の知見。

### Step 3: 修正（サブエージェント並列、ただし**ファイル単位で分割**）

**Finding 単位ではなくファイル単位**で `implementer` サブエージェントに割り当てる。同一ファイルへの並列書き込みは Edit 競合・整合崩れを起こす。

起動チェックで確認したファイル一覧をベースに、**1 ファイル 1 エージェント** で割り当てる。例（e-station 標準構成の場合）:

| 修正エージェント割当 | 担当 |
|---|---|
| ① 各ドキュメントファイル担当（複数） | そのファイルに関連する Finding を全部集約 |
| 最後の 1 体 | `review-fixes-<日付>.md` への当ラウンドのログ追記 |

> ファイル構成は `plan_area` によって異なる。起動チェックの `ls` 結果を見て割当を決めること。standard 構成（README / spec / architecture / implementation-plan / open-questions）であれば 5 体に分割する。

**各 implementer への指示テンプレ**:

```
docs/<area>/<FILE>.md を以下の Finding に従って修正する。
**<FILE>.md のみ書き換え**。他は Read のみ可。

【統一決定（全エージェント共通）】
- ＜オーケストレーターが Step 2 で決めた方針を 1 行ずつ＞

【Findings】
- **<ID> (HIGH/MEDIUM/LOW) <タイトル>**: <詳細> / <推奨修正>
- ...

修正後は対象節を Read で確認し反映を verify。
最後に「<FILE> 修正完了 / 反映節: ...」とだけ報告。
```

**ログ更新エージェント（⑤）の責務**:
- `review-fixes-<日付>.md` 末尾に「## ラウンド N（日付）」セクションを追記
- 全 Finding を表形式（Finding ID / 観点 / 対象ファイル:行 / 修正概要）で記録
- 当ラウンドで decided な統一方針も冒頭に箇条書き

### Step 4: 修正の検証

サブエージェント完了通知を受け取ったあと、**修正済みの宣言を信用しすぎない**:

- 用語統一のような横串修正の場合、**`Grep` で残存件数を機械的に確認**（例: `Grep "order_number"` で旧用語が残っていないか）
- 章節リンクのアンカー死活は、修正後の md ファイルから再度 `Grep "spec.md#"` でリンクパターンを抜き出し、対応する見出しが存在するか抜き打ち確認
- これらは費用対効果が高いので**ラウンドごとに 1〜2 回**走らせる

### Step 5: 次ラウンドへ

Step 1 に戻る。次ラウンドのレビュー指示には:

- **当ラウンドで修正された箇所を重点検査**する旨を明記
- 統一方針（前ラウンド決定）を冒頭に再掲し、その方針が他箇所に整合的に波及しているかを観点 A に追加検査させる
- ラウンド数が増えたら観点を統合（例: 「観点 A+B+D を 1 エージェントで合算」）して並列度を下げる

## 出力形式（毎ラウンド）

各ラウンド開始時:

```
=== ラウンド N ===
残存 HIGH: X件 / MEDIUM: Y件 / LOW: Z件

【統一決定】
- 決定 1: ...
- 決定 2: ...
```

各 implementer 起動時はサブエージェントが「修正完了 / 反映箇所」を報告するので、オーケストレーターは要点だけ転記。

`review-fixes-<日付>.md` 末尾に **「## ラウンド N（日付）」** ブロックを追記し続ける:

- 全 Finding を表形式（Finding ID / 観点 / 対象ファイル:行 / 修正概要）で記録
- 当ラウンドの統一決定を冒頭に箇条書き
- **サイズ管理**: 前ラウンドで解消した項目は次ラウンド以降で繰り返さない

## 完了サマリ

```
=== 完了 ===
全ラウンド数: N
修正した Finding 総数: HIGH X / MEDIUM Y / LOW Z
残存 LOW（対応不要）: K件
主要な反映成果（規約レベル）:
- 用語: ...
- エラーコード: ...
- 不変条件: ...
- 設計規約: ...
- テスト観点: ...
ログ: docs/<area>/review-fixes-<日付>.md
```

## ループ上限と escape hatch

- **最大 N ラウンド = 8** をハード上限とする。それを超えても収束しない場合は強制終了し、残存 HIGH/MEDIUM を `open-questions.md` に **未決オープン質問として書き出す**
- HIGH/MEDIUM 件数が **3 ラウンド連続で減らない**場合、観点設計が間違っているサインなのでユーザーに「観点を再設計するか、別アプローチに切り替えるか」を相談する

## オーケストレーター運用 Tips

### 並列起動

> 独立タスクは **同一メッセージ内で複数 Agent 呼出**。「4 観点並列」＝ 1 メッセージで 4 ツール呼出。順次起動するとコストも時間も無駄。

### バックグラウンド実行

レビューエージェントは長時間（数十秒〜数分）かかるため `run_in_background: true` で投入し、完了通知を待つ。Sleep ループは禁止。

### 修正範囲の判断

| 発見 | 対応 |
|---|---|
| HIGH（構造的矛盾・実装フェーズに影響） | 同ラウンドで修正必須 |
| HIGH（別フェーズの設計変更が前提） | **ユーザーに承認を取る**。承認後に `open-questions.md`「繰越」に明示 |
| MEDIUM | 同ラウンドで修正（このスキルの停止条件） |
| LOW | 列挙のみ。次フェーズで拾うかどうかをユーザーに判断させる |

### ラウンド終盤の観点合算

中盤（ラウンド 3 以降）は HIGH/MEDIUM が減り、全観点を 4 エージェントで回すのはコスト過多になる。**観点 A+B+D を 1 エージェントで合算 / C を単独**のように統合して並列度を下げる。

### 観点と禁止事項の参照先

観点 A〜D の詳細・禁止事項は `.claude/skills/e-station-review/SKILL.md` の「計画文書レビュー観点」「計画文書レビューの禁止事項」セクションを参照。ループ固有の追加制約のみ本ファイルで管理する。

## 禁止事項と失敗パターン

### ループ固有の禁止事項

- 同一ファイルに複数 implementer を並列で書き込ませてはいけない（必ずファイル単位で分割）
- 統一方針を Step 2 で決めずに implementer を並列起動してはいけない（再分裂の主因）
- レビューサブエージェントが「収束」と返した瞬間にループ終了してはいけない。**最終ラウンドだけは grep ベースの機械検証を 1 セット必ず追加する**

### 失敗パターン（避けること）

1. **MEDIUM を無視して「完了」にする** — ループ条件違反。MEDIUM ゼロまで繰り返す
2. **修正後の再レビューをスキップ** — 修正で新規 MEDIUM が混入していないか必ず確認する
3. **4 エージェントを順次起動** — 並列が原則
4. **統一方針を決めずに修正を並列投入** — ラウンドをまたいで再分裂する最大の失敗要因
5. **修正ログ更新を最後にまとめる** — ラウンドごとに追記しないと、次のレビュアーが「何が解消済みか」を判断できない
6. **実装フェーズ由来の指摘（コード品質・型安全）をここで解消しようとする** — 計画書レビューのスコープ外

## 知見（実績ベース）

- **収束は線形には進まない**。修正で新しい記述が増えると新たな矛盾の余地が生まれる。HIGH/MEDIUM 件数は減るが、ゼロまで届くのに 5〜7 ラウンドかかることが多い
- **Finding 件数の推移を監視せよ**（ラウンド 1: 30〜40 件 / ラウンド 4: 0〜数件 が標準的な減衰）
- **ファイル単位並列・観点単位並列が鉄則**。Finding 単位の並列は競合事故が起きる
- **observation point（実行コマンド・テストファイル名）の明記**を観点 D で徹底すると、実装フェーズに入ってから「テストが書きづらい」事故が激減する
- **横串の波及修正（用語統一など）は 1 ラウンドで全ファイル同時修正**。複数ラウンドに分けない
- 中盤以降は**レビュー観点を合算**してエージェント数を絞ると体感速度が上がる（A+B+D 合算 / C 単独 など）

## 適用例

立花証券 T0 仕様フェーズ実測（計画書 5 ファイル・約 1200 行）:

| ラウンド | 投入観点 | HIGH | MEDIUM | 主な修正内容 |
|---|---|---|---|---|
| R1 初回 | 4 並列 | 12 | 24 | 用語統一・フロー欠落・エラーコード不統一 |
| R2 再レビュー | 4 並列 | 4 | 8 | アンカー切れ・命名揺れ |
| R3 再レビュー | 2 並列（A+D / B+C） | 0 | 2 | サンプルコード・フォールバック漏れ |
| R4 サニティ | 1 体（全観点合算） | 0 | 0 | **収束** |

総所要: レビュー 11 並列起動 + 修正 3 ラウンド。繰越 0 件。

## 汎用呼び出しテンプレート

新フェーズの計画書レビュー開始時にオーケストレーター（あなた）に貼り付けて使う。`{{}}` プレースホルダーを実値に置換すること。

---

あなたは **オーケストレーター** です。`{{repo_name}}` の計画文書群 `{{plan_area}}` を レビュー → 修正のループで仕上げてください。

**唯一のリファレンス**: すべての手順・不可侵ルール・収束基準・既知の落とし穴は [`SKILL.md`](./SKILL.md) と本ファイル（[`PlanLoop.md`](./PlanLoop.md)）に集約されています。

```text

docs/{{plan_area}}/architecture.md
docs/{{plan_area}}/implementation-plan.md
docs/{{plan_area}}/open-questions.md
```


### 開始手順

1. 上記対象ドキュメントをすべて読み、現状を把握する
2. `PlanLoop.md` の「起動チェック」を実施してから「Step 1」へ
3. 各ラウンドの集約・修正・再レビューは本ファイルの手順に従う
4. **MEDIUM 以上ゼロ** で終了。完了後にユーザーへ最終サマリ（ラウンド毎の件数推移・繰越項目）を報告する

このスキル自体も品質収束する。新フェーズで適用した後、新しい知見が出たら本ファイルの「知見（実績ベース）」セクションに追記する。
