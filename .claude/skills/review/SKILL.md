# Review

計画書レビューでは、文章の見た目よりも **「このまま実装者が動いたときに事故るか」** を優先して見る。

## いつ使うか

- `implementation-plan.md` をレビューするとき
- `spec.md` / `architecture.md` / `open-questions.md` の整合を確認するとき
- フェーズ着手前に「この計画で実装して大丈夫か」を見たいとき
- 実装中のフェーズで、次のタスクへ進んでよいかを確認したいとき

## 最初にやること

最低でも以下を読む:

- 対象の計画書
- 同階層の `spec.md`
- 同階層の `architecture.md`
- 同階層の `open-questions.md`

計画書だけで閉じず、**相互参照先の現行文面まで確認する**。レビューは単一ファイルの校正ではなく、文書群の整合確認。

## 重点観点

### 1. 用語と wire 形式の一貫性

- 同じ概念が別の表記で書かれていないか
- enum / schema / IPC 文字列が途中で変わっていないか
- UI 表示名と内部値を混同していないか

典型例:

- `D1` と `1d` が混在する
- `display_symbol` と `display_name_en` が混在する
- `VenueError` と旧 `EngineError` が併記される

### 2. 先に確定した方針を後段で覆していないか

- T0 で「実装位置を確定」と書いたのに、T4/T5 で別の場所に再配置していないか
- open question が「決定済み」なのに、後段のタスクで未決のように扱っていないか

見るべきもの:

- 「決定」「確定」「閉じる」「正本」「唯一」の語
- 「実装位置」「責務」「source of truth」

### 3. ゲート条件とフェーズ順序

- 「T5 着手前に必須」と「T1 着手前に必須」が混在していないか
- ブロッカーが未解決なのに受け入れ条件だけ先に進んでいないか
- フェーズの依存関係が本文とリスク表でズレていないか

レビューでは **未解決ブロッカーの存在そのもの** より、**どこまで進んでよいかの線引きが一貫しているか** を重視する。

### 4. 受け入れ条件の実装可能性

- 受け入れ条件が曖昧語で終わっていないか
- テスト名、入力、期待結果があるか
- ネガティブテストが必要なのに抜けていないか
- 「UI が落ちない」「重複再送しない」など、壊れやすい性質に pin テストがあるか

危ない表現:

- 「適切に処理する」
- 「必要なら」
- 「後で確認」
- 「問題なければ」

### 5. 未定義フィールド・未定義データ経路

- 途中から新しい key / field / map 名が出ていないか
- 「送る」と書いてあるのに schema / DTO / event shape に載っていないものがないか
- 「別管理 map」と書いてあるのに、その生成タイミングと所有者が書かれているか

### 6. 実装者が誤読しやすい箇所

以下は要注意:

- 肯定形と否定形が近接している箇所
- 旧方針の否定が長文に埋もれている箇所
- 同じ段落で「不要」と「必須」が混在する箇所
- 例外ルールが脚注だけにある箇所

レビューでは「正しいか」だけでなく、**急いで読む実装者がどう誤解するか** を考える。

### 7. 両側性 — Rust ↔ Python のタスク対称性

フェーズが Rust と Python の両側にまたがる場合、片側のタスクだけ細かく書かれて反対側がスカスカになりがち。**完了条件から逆算して、両側に対応する作業項目があるか**を必ず俯瞰する。

確認項目:

- IPC で送られる DTO / event を **送信側 (Python) と受信側 (Rust)** の両方に配線タスクがあるか
- Rust 側の DTO / 構造体 (`engine-client/src/backend.rs` の受信ハンドラ等) に「受信した dict をどの型に詰め直すか」が書かれているか
- 受信側で値が落ちる経路（`new()` で `None` 上書き、`new_stock()` を使わない、normalize を呼ばない、別 map に格納されない等）がレビュー対象か

過去の事故: T4 で Python 側 `list_tickers` / `fetch_ticker_stats` だけ書かれており、Rust 側 `TickerMetadataMap` 構築箇所が `TickerInfo::new(...)` 経由で `display_name_ja` / `lot_size` が落ちる事象を見逃した。

### 8. ライフサイクル全体 — happy path 以外を網羅する

「初回起動 → 1 回ロード → 表示」だけ見ていると、再起動・再ログイン・状態切替・日跨ぎでの **reset / invalidation 規約** が抜ける。

確認項目（ライフサイクルチェックリスト）:

- **再起動**: プロセス再起動時にメモリ状態・キャッシュ・カウンタは何が消え何が残るか
- **再ログイン / 再 ready**: idempotent イベント（`VenueReady` 等）の再受信時に重複処理が起きないか、ProcessManager 側で集約されるか
- **状態切替**: env / mode (demo↔prod) が変わった時に古い状態（マスタ・session・URL）が残らないか
- **時刻境界**: JST 日跨ぎ・市場営業日切替・session 期限到達時の挙動
- **長期稼働**: プロセスが何日も生きた時の蓄積（counter overflow・メモリ単調増加）

過去の事故: マスタ in-memory invalidation 規約（demo/prod 切替・JST 日跨ぎ・`__init__` 再生成）が完全に抜けていた。

### 9. 複数入口の網羅 — UI ガードだけでは止まらない

ある機能への入口は通常 1 つではない。「UI で非活性化する」だけでは、**他の経路から同じリクエストが flight する可能性**を見落とす。

確認項目:

- **保存済み state 復元経路** (`saved-state.json` / `pane.json` 等) は capabilities 適用前にも flight する
- **capabilities 受信前の race**: `Ready` 受信から capabilities 適用までの window で初期 subscribe が走らないか
- **直接 IPC** (テスト・スクリプト・別 venue 経由) からの呼出
- **再接続後の自動再購読** が古い設定で走らないか

各入口に対して **backend 側で fail-safe（明示的拒否 + テスト pin）** が必要か検討する。UI ガードだけを「守られている」と扱わない。

過去の事故: 非 `"1d"` kline 要求を「UI 非活性化で防ぐ」だけにしており、saved-state 復元経路から flight する可能性を見落とした。

## 典型的な指摘パターン

- **矛盾**: 同じ文書内、または関連文書間で方針が食い違う
- **未定義**: 後段で使う名前や field が、前段で定義されていない
- **逆流**: 旧名称・旧 DTO・旧 error code が一部に残っている
- **順序破綻**: 先に必要な前提が未確定のまま後続タスクが始められる
- **テスト不足**: ポジティブしかなく、誤判定や silent failure を防げない
- **責務曖昧**: Rust / Python / UI / backend のどこが正本か分からない

## 出力のしかた

コードレビューと同じく、**findings first**。

各 finding には以下を入れる:

- `path:line`
- 何が矛盾 / 不足 / 曖昧か
- なぜ危険か
- 可能なら「どちらに揃えるべきか」の短い示唆

優先順位:

1. 実装を誤らせる矛盾
2. ゲートや受け入れ条件の破綻
3. 仕様未定義や命名不整合
4. テストや運用上の抜け

## 実務 tips

- まず `rg -n "決定|確定|正本|唯一|禁止|必須|受け入れ|TODO|Phase"` で強い文言を拾う
- 次に固有名詞を横断検索する
- 表やリスク欄は本文とズレやすいので必ず見る
- 「完了根拠」「レビュー反映」ブロックは信用しすぎず、現在の本文と照合する
- 1 件見つけたら、その用語の全出現箇所を追う
- 文章の改善提案より先に、実装事故に直結する点を出す

## 修正提案を書く側に回ったときの自己チェック（最重要）

レビューで Finding を上げるだけでなく **修正タスクを起草するとき**は、自分が新たな矛盾を仕込まないために以下を必ず実行する。これを飛ばすと **次ラウンドで自分の修正が指摘の的になる**。

### A. 既存確定方針への再 grep（書いた直後）

修正タスクを書き終えたら、その本文に登場する **固有名詞・型名・wire 値・フィールド名** を全部 grep して、既存確定記述と一致しているかを機械確認する。

- wire 形式 (`"1d"` vs `D1` 等) — T0/T0.2 で確定した serde 形式と一致しているか
- 責務の所在 ("正規化は saved-state ロード時のみ" 等) — 確定済みの単一責務を覆していないか
- 確定済みフィールド名 (`display_symbol` / `display_name_ja` 等) — 勝手に新名 (`display_name_en` 等) に置換していないか
- ゲート対象フェーズ ("T5 着手前必須" 等) — 本文・リスク表・inventory で一致しているか

### B. 参照する関数・クラス・ヘルパの存在性 grep

修正タスクで「`Foo.bar()` で X を検知し」と書いたら、`Foo` クラスと `bar()` メソッドが **実コードに存在するか** を grep で確認する。

- 関数・クラス・モジュールが実装済みか（同フェーズで新設なら「新設するタスク」が先に必要）
- 既存ヘルパの **戻り値型** が想定どおりか (`current_p_sd_date()` がミリ秒精度値か日付文字列か等)
- 直接呼ぼうとしている setter / getter / フィールドが実在するか

過去の事故: `TachibanaWorker.set_credentials(creds)` を前提にしたが `TachibanaWorker` クラス自体が未実装。`current_p_sd_date()` を「日付として比較」と書いたが実際はミリ秒精度タイムスタンプ。

### C. 概念書換時の波及 grep

ある文書で用語・型名・規約を書き換えたら、**他文書（特に open-questions / invariant-tests / README / SKILL）で同じ概念の旧表記が残っていないか** を必ず grep する。

- 旧型名 (`TickerListed` 等) を `TickerInfo.tickers[*]` に変えたら、Q&A・決定済み欄・設計メモ全部を grep
- 修正提案で test 関数名を新規命名したら、**invariant-tests.md の対応表に追記**するタスクを必ずセットで含める（これを忘れると CI grep ガードが warning を量産する）
- 行番号参照を「シンボル参照」に切替えたら、同方針を全文書で grep

レビューと修正は **対**で動かす。修正の質が次のレビューラウンドの母集団を決める。

## ひとことで言うと

良い計画書レビューは「きれいな文章」にすることではなく、**実装者が間違っても不思議でない曖昧さや矛盾を先回りで潰すこと**。
